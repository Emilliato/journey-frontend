How it works

JOURNEY keeps working with no internet.

When there's no connection, JOURNEY doesn't go quiet. A small AI model loads inside the browser and answers your child directly on their device — then quietly syncs everything back when the connection returns. Here's the whole picture.

Llama 3.2 1-billion-parameter model ~880 MB runs on the phone's GPU 0 data leaves the device offline
1

Two brains, one companion

JOURNEY chooses where to think based on a real connectivity check — not the browser's unreliable "online" flag, but an actual ping to the backend. Online, it uses the powerful cloud model. Offline, it switches to the on-device model. Your child just keeps chatting.

🔌 Connectivity check
A lightweight health-check decides the route on every message
Online
Cloud brain — the message goes to the backend Claude proxy and back. Full-strength answers.
Offline
On-device brain — the local model replies right on the phone. No server, no signal needed.
🔀

If the signal drops mid-conversation, the running chat is handed to the on-device model with its history intact — your child continues the same conversation without noticing the switch.

2

Getting the model onto the phone

The model is downloaded once, ahead of time — while there's still a connection — then cached in the browser so it's ready the moment it's needed. Each step depends on the one before it.

Step 1 · App opens
Warm start
The download begins quietly in the background the instant the app loads — no waiting for your child to go offline first.
Step 2 · Download
~880 MB of weights
The model files stream down over any available connection. This is the one big cost, and it happens only once.
Step 3 · Cache
Stored in the browser
The weights are kept on the device, so future sessions — and every offline session — skip the download entirely.
Step 4 · Warm up
Compile for the GPU
A tiny throwaway run compiles the GPU shaders ahead of time, so the first real reply isn't slowed by one-time setup.
Step 5 · Ready
Offline-capable ✓
From here JOURNEY can answer with no connection. A "Check offline readiness" button lets you confirm this in advance.
3

Anatomy of one offline reply

A small model can't be trusted to know everything, so it isn't asked to. Before it generates, JOURNEY assembles a focused prompt from what it already has on the device — then streams the answer back word by word.

💬 Your child types a message
🧩 Assemble a focused prompt
Tight persona & rules What JOURNEY remembers about the learner Relevant notes from a built-in library The learner's saved goals Last few chat turns
⚙️ The model runs on the GPU
Inside a background worker, capped short (~120 words) so replies finish quickly even on a mid-range phone.
✨ Streams into the chat
Words appear as they're generated — the reply starts almost immediately instead of after the whole answer is done.
🛟

Honesty over guessing. The persona tells the model to give one clear, correct step — and if it isn't sure, to say "let's check that one when we're back online" rather than invent a fact. For a child, a wrong answer is worse than a deferred one.

4

Why the app stays smooth

Running an AI model is heavy work. If it shared the same thread that draws the screen, the app would freeze while it thought. So the model lives in a separate background worker — the screen stays responsive the whole time.

🖐️ Main thread stays free
  • Draws the chat, avatar and animations
  • Handles taps, typing and scrolling
  • Never blocks while the model thinks
⚙️ Background worker
  • Loads and holds the model
  • Runs the GPU computation
  • Sends finished words back to the screen
📴

The device's GPU is reserved for the model. That's why the whole interface uses lightweight drawing (simple shapes and gentle motion) and no heavy graphics — the graphics budget belongs to the AI.

5

Safe offline, synced on return

Anything JOURNEY learns offline — a new goal, a noted preference — is written locally and marked "to sync". The moment the connection is back, it's reconciled with the server. And it's all still gated by the same rule as online: nothing about a learner is recorded without your active consent.

Offline
New goals & memories are saved on the device and flagged pending sync — but only if consent is active.
📶 Connection returns
Detected automatically
Synced
Queued changes are pushed to the server and the local copies are reconciled. The status shows right next to the connection pill.
6

What a device needs

Offline JOURNEY needs a phone whose browser can run computation on the GPU (WebGPU). Most phones from the last few years qualify. Where they don't, JOURNEY degrades gracefully rather than breaking.

Browser
Chrome 113+
Or any browser with WebGPU enabled
GPU memory
~0.9–1.1 GB
For the model while it runs
One-time download
~880 MB
Cached after the first fetch
Storage
~1 GB free
To keep the cached model
🪶

No WebGPU? It still works. On a device that can't run the model, JOURNEY drops to a cached-content mode: your child still sees their profile, goals and saved chat, and can reconnect to use the cloud brain. The app never simply fails.

In one line: a small AI model is downloaded once, cached on the phone, and run inside a background worker on the device's GPU — so JOURNEY keeps helping your child offline, honestly and safely, and syncs back the moment there's a signal.
